# Writing a StriVo plugin

This guide walks the full path from `mkdir crates/your-plugin/` to a
shipping marketplace entry. Companion to
[`docs/PLUGIN-MANIFEST.md`](PLUGIN-MANIFEST.md) which is the
reference for the manifest spec itself.

## The shape of a StriVo plugin

Every shipping plugin in `crates/<name>/` is a **pure-data Rust
crate**: typed models + functions over them, no IO, no DOM, no
ffmpeg spawn. The host (`crates/strivo-web/`) is the only place
that talks to the network, the database, ffmpeg, or the user.

This split keeps the plugin trivially unit-testable, lets the same
crate run in three contexts (host backend, marketplace catalogue,
future WASM sandbox), and means every "what does this plugin
actually do" question has a single-file answer.

A plugin produces some combination of:

- **A typed model** the user can edit (an `InsertChain`, a
  `RecipeChain`, a `PitchTime`).
- **Composers / parsers** that turn raw output (ffmpeg stderr, a
  chat log, an EDL) into structured data, or back the other way.
- **Pure transforms** — score this bucket, snap that timecode,
  build this filtergraph.

## Directory layout

```
crates/your-plugin/
├── Cargo.toml          # package + deps
├── src/lib.rs          # everything
└── README.md           # autogenerated by scripts/gen-plugin-readmes.sh
```

That's it. No `build.rs`, no `tests/` dir (unit tests live under
`#[cfg(test)] mod tests` in `lib.rs`), no example binaries. The
marketplace + capability matrix + host wiring all live elsewhere.

## Cargo.toml skeleton

```toml
[package]
name = "strivo-your-plugin"
version = "0.1.0"
edition = "2021"
description = "One-line description that lands verbatim in the marketplace card."
authors = ["you <you@example.com>"]
license = "MIT"

[lib]
path = "src/lib.rs"

[dependencies]
serde = { version = "1", features = ["derive"] }
# Add only what you genuinely need. Heavy crates (reqwest, tokio,
# rusqlite) signal that work should move to the host instead.

[dev-dependencies]
serde_json = "1"
```

## The typed model + tests

Start with the smallest serialisable struct that captures what the
user is editing. For the loudness plugin that was

```rust
pub struct LoudnessTarget {
    pub i: f64,
    pub tp: f64,
    pub lra: f64,
}
```

…then everything else was a pure function over `LoudnessTarget`
(parser, delta calculator, ffmpeg filter builder). Tests come next:

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn round_trips_through_serde() {
        let t = LoudnessTarget { i: -14.0, tp: -1.0, lra: 7.0 };
        let s = serde_json::to_string(&t).unwrap();
        let back: LoudnessTarget = serde_json::from_str(&s).unwrap();
        assert_eq!(back.i, -14.0);
    }
}
```

The bar is *unit-testable without ffmpeg, without an HTTP client,
without a temp dir*. If you find yourself wanting a fixture file or
a mocked subprocess, the work belongs in the host, not in the
plugin.

## Capability tags

Each plugin declares one or more capability tags in its marketplace
manifest. The well-known list lives in
`strivo_core::plugin::capability`; DAW-vision extensions use the
`x.`-prefixed namespace (`x.sidechain`, `x.pitch_time`,
`x.insert_fx`). New tags get added to `crates/marketplace/` +
`crates/strivo-web/src/routes/api.rs` so the capability matrix
picks them up.

## Wiring into the host

The host wires every plugin via three steps:

1. **Workspace member.** Add `"crates/your-plugin"` to the
   `[workspace] members` list at the repo root.
2. **Dependency.** Add the path dep to `crates/strivo-web/Cargo.toml`
   so the backend can call into it.
3. **Route handler.** Add a route in
   `crates/strivo-web/src/routes/plugins.rs` that calls the crate's
   pure functions, pipes the result through ffmpeg / persistence /
   whatever the host needs, and serves it as JSON.

The marketplace catalogue entry in `crates/marketplace/src/lib.rs`
holds the user-facing description, capability tags, price (or
`None` for free), and `installed` flag.

## SPA wiring

The SPA expects:

- An entry in `PLUGIN_REGISTRY` at the top of `assets/spa.js`.
- API client method(s) under the `API` object.
- A renderer or panel for the plugin's UI surface. Look at
  iter-50–53 commits for the canonical "small modal panel +
  editor topbar button" pattern.

Existing examples worth reading:

- **iter-53 pitch** — typed model + clamp/round-trip tests +
  inline editor panel with sliders.
- **iter-52 insert-fx** — enum of effect variants, each mapping
  to one ffmpeg filter, chain composes left-to-right.
- **iter-50 sidechain** — composes VAD output (another plugin)
  into a volume-automation curve (a third plugin); demonstrates
  cross-plugin composition with no new IO at the boundary.

## Shipping

When the plugin lands on `main`:

- `scripts/gen-plugin-readmes.sh` synthesises a top-level README
  from the crate metadata.
- The marketplace catalogue exposes it under the right Editor /
  Publish / Viewer / Analytics category.
- The capability matrix at `/plugins` picks up the new
  `x.<tag>` rows automatically.
